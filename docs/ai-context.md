# AI Context

## Purpose

Worker Blog 是一个 Cloudflare-first 的 CMS/blog 项目。它提供：

- Public API: 面向公开站点或 headless consumer 的内容、媒体、事件接口。
- Admin API: 面向管理后台的内容、媒体、分类、标签、日志、安全审计、分析、设置、个人资料接口。
- Admin UI: React/Vite 单页管理后台。
- Editor: 共享 Tiptap 富文本编辑器包。
- Shared: 跨 server/admin/editor 的 TypeScript 类型、zod schema、API contract 和纯工具函数。

## Package Map

| Package | Role | Depends On |
| --- | --- | --- |
| `packages/server` | Cloudflare Worker 后端，Hono 路由，D1/R2/KV 访问，静态 Admin assets fallback | `@worker-blog/shared`, `@worker-blog/admin`, `@worker-blog/editor` |

| `packages/shared` | Contracts, schemas, shared types, utilities | `zod`; currently imports `JSONContent` type from editor in content contract |

## Runtime Request Flow

1. Cloudflare Worker starts from `packages/server/src/index.ts`.
2. `createWorkerBlogApp()` creates a Hono app in `packages/server/src/app.ts`.
3. `packages/server/src/app-registration.ts` registers middleware and routes.
4. API requests under `/api/*` run through Hono.
5. Media files under `/files/*` stream from R2.
6. Non-API GET requests fall back to Worker Assets from `packages/admin/dist`, serving the Admin SPA.

## Most Important Design Decisions

- `packages/shared` owns shared contracts. Do not duplicate request/response schemas in admin/server.
- Initial setup is guarded: most `/api` traffic returns `428 SETUP_REQUIRED` until the first admin exists.
- Browser auth uses `auth_token` HTTP-only cookie plus `csrf_token` readable cookie. Mutating browser requests send `X-CSRF-Token`.
- Bruno/API token usage can use `Authorization: Bearer <token>` and does not need CSRF.
- Admin routes are authenticated and role-gated. Default `/api/admin/*` guard allows admin only unless overridden by route-level or app config.
- Content source of truth is `contents.body_json` / `bodyJson` Tiptap JSON.
- Published rendering cache is `contents.body_html` / `bodyHtml`; it is generated by server/editor rendering and sanitized. Do not accept client-provided `bodyHtml` in write contracts.
- Existing migration files should not be edited.

## Known Integration Notes

- `packages/server/src/features/user-profiles/index.ts` implements `/schema`, `/:userId` GET, and `/:userId` PUT routes, and Admin client calls `/api/user-profiles/*`.
- Current `packages/server/src/app-registration.ts` does not mount the user-profiles feature route. Treat this as an integration gap to verify before relying on those endpoints.
- `packages/shared/src/schemas/index.ts` is currently a placeholder used by `/api/health`.
