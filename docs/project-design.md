# Project Design

## Product Goal

Worker Blog aims to be a compact Cloudflare-native CMS/blog system:

- Small enough to run as one Worker-backed deployment.
- Structured enough to support admin workflows, public content APIs, media storage, audit logs, and analytics.
- Friendly to AI-assisted development through explicit shared contracts and package boundaries.

## Primary Users

| User | Needs |
| --- | --- |
| Site owner/admin | Set up first account, manage content, upload media, configure site/security settings |
| Editor/author | Draft and publish content, organize categories/tags, manage rich-text body |
| Public consumer | Read published content through API, load public media files |
| Operator/developer | Run locally, inspect logs, audit auth events, deploy to Cloudflare |

## Core Product Modules

### Setup and Auth

The system starts locked until a first admin exists. This avoids shipping a public registration system by default.

Design choices:

- First registration creates an admin.
- Later registration is disabled.
- Login returns JWT and sets browser cookies.
- Browser mutation protection uses CSRF.
- API tools can use bearer JWT.

### Content Management

Content management is centered on structured editing and safe public rendering.

Design choices:

- Admin edits Tiptap JSON.
- Server stores `bodyJson` as canonical content.
- Server renders sanitized `bodyHtml` for published output.
- Admin can view version history and restore snapshots.
- Delete from Admin is soft delete.
- Slugs are unique among non-deleted content.

### Taxonomy

Categories and tags provide organization:

- Categories can be hierarchical through `parent_id`.
- Tags are flat and carry a color.
- Content can have one category and many tags.
- Category/tag deletion is blocked when in use.

### Media Library

Media library separates metadata and bytes:

- Bytes are stored in R2.
- Metadata is stored in D1.
- Duplicate active uploads are detected by SHA-256 hash.
- Public access goes through `/files/*`.
- Content tracks body and cover references to media.

### Observability

The project includes operational views for:

- Dashboard stats.
- System logs.
- Log config.
- Security audit.
- Analytics events.

Some observability tables may be absent during first boot/tests, so routes generally degrade gracefully.

## Non-Goals For Current Shape

- This is not a multi-tenant system yet.
- This is not a public user registration product.
- This is not an e-commerce or membership platform.
- This is not currently designed around backward compatibility because the project has not launched.

## Design Principles

1. Prefer Cloudflare-native primitives: Workers, D1, R2, KV, Worker Assets.
2. Keep shared contracts in `packages/shared`.
3. Keep runtime-specific logic in `packages/server`.
4. Keep Admin presentation in `packages/admin`.
5. Keep rich editor concerns in `packages/editor`.
6. Treat editable content and rendered content as separate representations.
7. Store source data, derive caches.
8. Make setup and auth secure by default.
9. Preserve clear AI-readable paths and documentation.

## Data Ownership Summary

| Data | Owner |
| --- | --- |
| Auth/session cookies | Server middleware/routes |
| User records | D1 `users` |
| Custom profile data | D1 `user_profiles` and user-profile feature |
| Content source | D1 `contents.body_json` |
| Public rendered content | D1 `contents.body_html` generated by server/editor renderer |
| Media bytes | R2 `MEDIA_BUCKET` |
| Media metadata | D1 `media` |
| Cache and lockouts | KV `CACHE_KV` |
| API contracts | `packages/shared/src/admin-api/` |

## Future Design Questions

- Should user profile feature routes be mounted under `/api/user-profiles` in app registration?
- Should public content support lookup by slug in addition to ID?
- Should analytics/security-audit schemas be consolidated into Drizzle schema source?
- Should root `pnpm type-check` remain manually enumerated or switch to recursive package scripts?
- Should Admin role boundaries distinguish author/editor permissions more strictly per content owner?
